# Post-PR Creation Tracking & Feedback Handling Guide

Use this guide following Phase 4 of the `ship-it` workflow to submit the Pull Request in draft mode, monitor CI/CD checks, process initial code review feedback, and mark the PR as ready for review.

---

## 1. Step-by-Step Execution

### Step 1: Prompt User for Draft PR Submission Approval
Before running any `git push` or `gh pr create` commands:
1. Present the final PR title and description (generated by the `write-pr-description` skill) to the user.
2. Use `ask_question` to request explicit user confirmation to submit the draft PR:
   - "Are you ready to push changes and create the Draft Pull Request on GitHub?"

### Step 2: Submit Draft PR via GitHub CLI
Once approved:
1. Push branch to remote: `git push -u origin <branch-name>`
2. Create PR in **Draft Mode** (`--draft`) using `gh pr create` (following the `gh-cli` skill):
   ```bash
   gh pr create --draft --title "<PR Title>" --body-file <path-to-pr-description-artifact>
   ```

---

## 2. CI Monitoring & Schedule (5-Minute Check)

1. **Schedule Monitoring Timer**:
   - Use the `schedule` tool to set a 5-minute timer (`DurationSeconds="300"`, `Prompt="Check draft PR status for CI failures and code review feedback"`).
2. **Inspect Status on Expiry**:
   - Run `gh pr checks` to check CI pipeline status.
   - Run the `analyze-github-pr` script to fetch unresolved review comments in JSON format:
     ```bash
     env -u GITHUB_TOKEN python3 ~/.gemini/config/skills/analyze-github-pr/scripts/analyze_comments.py --json --dir <workspace-dir>
     ```

---

## 3. Feedback Resolution & Marking Ready for Review

If CI checks fail or review comments are found on the PR:

1. **Prompt User via `ask_question`**:
   - "CI check failures or review comments were detected on your draft PR. How would you like to handle them?"
   - **Option A (Interactive Dashboard)**: "Use the interactive browser dashboard to review and approve proposed fixes (`pr-feedback-handler` skill)."
   - **Option B (Direct Fix & Respond)**: "Automatically fix code findings, verify with unit tests, and post thread replies directly."

2. **Option A: Interactive Dashboard Workflow**:
   - Delegate execution to the `pr-feedback-handler` skill.
   - Enrich comment JSON with `proposedFix` and `draftReply` fields.
   - Launch `launch_dashboard.py` and wait for user approval before implementing changes.

3. **Option B: Direct Fix & Response Workflow**:
   - Implement recommended fixes directly in the working tree.
   - Run project linter and unit tests to verify fix accuracy.
   - Commit changes: `git commit -m "fix(pr): address review feedback"`.
   - Push to remote branch: `git push`.
   - Use `pr-feedback-handler` helper scripts to submit thread replies and resolve threads:
     ```bash
     python3 ~/.gemini/config/skills/pr-feedback-handler/scripts/update_thread.py --file <feedback-state-json>
     ```

4. **Mark PR Ready for Review**:
   - Once all CI checks pass cleanly and initial review feedback is resolved, prompt the user via `ask_question`: "All CI checks pass and initial feedback is resolved. Mark PR as ready for review?"
   - Upon approval, run `gh pr ready` to publish the PR out of draft state.
